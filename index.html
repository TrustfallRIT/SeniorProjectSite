Project Synopsis

The purpose of the project is to automate employer and student interactions during career fairs. This includes signing up for and signing into interviews. Employers will be able to create as many schedules as they need depending on how many interviewers they decide to bring to the interview day. After deciding which students they will interview, employers will be able to schedule the students on any of their schedules as long as the student is available during the chosen interview block. Each student will also have their own schedule which will let them view and check into interviews. When a student checks into an interview, the interviewer gets notified via a change in the status of the interview object on their schedule. The student's schedule view will also help the student know where to go for their interview by including details such as the location where they will be interviewed.

Development Methodology

Chosen Methodology

Agile Kanban is the selected development methodology based on the project's schedule and organization. Time is limited to complete this project, and assigning tickets every 2 weeks in a normal sprint cycle would be too slow and could allow features to fall through the cracks. A KanBan board still enables assigning points to tickets in order to accurately gauge progress week by week.

Scheduled Meetings

Meetings with group members will be held each Monday at 1:00pm. These meetings will be used to review group member's work and prepare artifacts to be submitted to the coach/sponsor. There will also be a weekly meeting with the sponsor and coach each Wednesday at 5:00pm. This will be to present artifacts and progress to the sponsor, ask questions, and generate feedback.

Planned Activities and Artifacts

Development will follow a cyclical process.

Key phases:

  • Planning: Defining scope, requirements gathering, risk assessment. This will be done on Wednesdays after meeting with our sponsor.
  • Design and Implementation: After our meeting on Wednesday, team members will complete tickets over the course of the week, either alone or with people that are working on the ticket with them from start to finish. Team members will not switch what tickets they are working on between Wednesday and Monday.
  • Testing: Team members will write testing for their code with easy to understand explanations of the tests. The tests will be performed and passed by a different member than the one who wrote the tests. Testing will be done after team meetings on Mondays, before the next sponsor meeting.
  • Deployment & Maintenance: Merges and deployment will be approved and managed by project lead Ian.

Product/Process Metrics

Process Metric: Estimation Accuracy

Reason: Since we are developing in a constrained time frame, tracking the accuracy of our time estimations for various tasks will help refine our team's planning and project management.

Measurement: Compare the estimated time for completing development tasks (from sprint planning or senior project timelines) against the actual time spent (tracked through time logs).

Goal: Reduce the gap between planned and actual effort to improve predictability and efficiency in future development cycles.

Product Metric: Number of Clicks Per Operation

Reason: A smooth interview scheduling process is a critical concern of the project. Since usability was a concern for the current product, creating an intuitive website with few nested submenus is essential for letting employers and students communicate correctly.

Measurement: Record and analyze the number of clicks it takes a user to successfully complete an action through the website UI. During UX testing, this will also include actual clicks (including incorrect clicks due to unintuitive design layout) as well as minimum clicks.

Goal: Reduce the number of clicks required to successfully complete website actions.

Major Deliverables and Milestones

First Milestone - Project Setup

The first milestone will involve completing everything required for project deployment to the RIT server and the development environment set up locally on all team members' devices. This milestone will conclude when the project API and database is running independently on the RIT servers, and when the project repository has a CI/CD pipeline running automated tests.

Second Milestone - Initial Testing & Project Direction

The second milestone will involve the development and implementation of initial unit tests, which will be run automatically by the CI/CD pipeline. It will also involve a finalization of the MVP and completion of the backlog. This milestone will conclude when the remaining features are planned and approved by the sponsor, and each code component has sufficient associated automated testing.

Third Milestone - Frontend Overhaul

The third milestone will conclude when the UI has been overhauled to have a consistent theme in line with RIT's branding and, most importantly, accessibility for mobile devices. A series of user tests will be conducted to ensure the UI is intuitive, and that all actions can be completed by each user type on both a desktop and mobile device. There may be several rounds of testing, followed by development, before this milestone is complete.

Fourth Milestone - Final Bug Fixes & Testing

The fourth milestone will conclude when criteria for the MVP has been reached, proven sufficiently by testing and approved by the sponsor. This milestone may also involve implementation of additional features not included in the MVP. It will also include a final set of stress tests.

Domain Model

Stakeholder Management Communication

Meetings

Meetings with the project sponsor will be held weekly to give updates on our progress, show and get feedback on implemented features, address blockers, and describe the plan for the coming week and beyond. Between these meetings, communication with the sponsor will be conducted asynchronously over email by the communications lead. This will be used to answer development questions and request access to resources as needed. The sponsor will also have full access to the team's project management board to track ticket progress in real time.

Scope & Features

Via user stories, the team will have a concrete outline of the MVP, which will be the central focus until it is complete. If needed, the MVP can be adjusted during the weekly sponsor meetings in light of unexpected bugs or unpredicted necessary features. In addition to the MVP, the team will maintain a backlog of features that would improve the application beyond the necessities. If time permits, the team will implement these features in order of importance dictated by the sponsor. If at any point one of these features is deemed necessary for the MVP, it can be added to the MVP but possibly in exchange for a different feature given development time restraints.
